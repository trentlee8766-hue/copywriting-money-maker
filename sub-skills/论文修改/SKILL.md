# 论文修改子技能

**所属：撰稿印钞机**
**定位：阶段二 - 论文修改执行引擎**
**适用场景：批注修改、降重、降AI、内容调整、综合修改**

---

## 核心职责

1. **修改类型判断**：识别 content_modify / content_add / content_delete / format_modify
2. **优先级排序**：基于内容性质制定处理顺序
3. **内容修改执行**：遍历段落按内容匹配后替换
4. **格式保护**：确保所有修改操作不破坏原始文档格式

---

## 四种操作类型定义

| 操作类型 | 定义 | 示例 |
|---------|------|------|
| content_modify | 现有内容的文字修改/换写 | "时间不对" → 改日期 |
| content_add | 新增内容到指定位置 | "需要补充研究背景" |
| content_delete | 删除无意义/错误内容 | "删除重复段落" |
| format_modify | 格式调整（非内容） | "三线表格式不对" |

**format_modify 处理原则**：
- 统一记录为"待格式处理"
- 不在内容skill中处理
- 交付时告知客户需要单独处理格式

---

## 操作处理顺序规范

**必须按此顺序处理**：

```
第一步：content_delete（删除）
  → 先腾出空间，避免删除影响后续新增内容的位置

第二步：content_add（新增）
  → 在删除后的结构上新增内容，位置准确

第三步：content_modify（修改/替换）
  → 处理剩余的文字修改和换写

第四步：format_modify（格式）
  → 统一记录，不在内容处理阶段处理
```

**为什么这个顺序**：
- 删除和新增会影响段落结构，先处理会导致后续步骤位置偏移
- 先删除腾空间，再新增内容，最后处理剩余修改
- format_modify 不影响内容结构，可最后统一记录

---

## 修改铁律

### 1. 格式保护第一
任何文本修改操作，必须保持原始文档格式不变。修改前后验证：
- 段落数一致
- 节数一致
- 表格数量一致
- 页眉页脚不变
- 标题样式不变

### 2. 按内容匹配，不按编号替换
**错误做法**：按段落编号硬替换（编号在用户手动编辑后会失效）
**正确做法**：遍历所有段落，按文本内容匹配目标段落后再替换

### 3. 先降AI再处理批注
优先级顺序：降AI → 批注处理 → 内容调整
原因：降AI会改动大量文本，若批注处理在前面，改完又被降AI打乱

### 4. 先复制副本，再修改副本
- 第一步：复制原文档得到副本（`Document('src.docx').save('副本.docx')`）
- 第二步：在副本上修改内容，原始文档保持不动
- 第三步：验证副本格式与原文档一致后，输出副本
- 严禁直接在原文档上修改
- 严禁用 `deepcopy` 复制段落到新 Document

## 各操作类型标准规范

### content_delete 操作标准

**适用范围**：删除错误内容、重复内容、无意义内容

**操作规范**：
1. 确认要删除的内容是否在章节标题中
   - 如果是章节标题 → 保留标题，只删除标题下的内容
   - 如果是正文内容 → 删除整段
2. 删除后检查：
   - 段落数是否正确（减少1个或多个）
   - 章节结构是否完整（标题仍在）
   - 页眉页脚是否正常
3. 删除操作不改变其他段落的内容和位置

**禁止行为**：
- 禁止删除章节标题（只删内容）
- 禁止删除页眉页脚
- 禁止删除表格（只能清空表格内容）

---

### content_add 操作标准

**适用范围**：在指定位置新增内容

**操作规范**：
1. 明确插入位置：
   - 找到目标段落（按内容匹配）
   - 在目标段落**之后**插入新段落
   - 如果要求在某个词之前，在匹配时记录位置
2. 插入后检查：
   - 总段落数是否正确（增加1个或多个）
   - 插入内容与上下文逻辑是否通顺
   - 段落样式是否与周围一致
3. 内容衔接：
   - 必要时添加过渡句
   - 检查新内容是否与前后段落主题相关

**特殊情况**：
- 如果要求在文档开头插入 → 在第一个段落之前插入
- 如果要求在文档结尾插入 → 在最后一个段落之后追加

---

### content_modify 操作标准

**适用范围**：文字修改、换写、降AI

**操作规范**：
1. 找到目标段落：
   - 遍历所有段落
   - 按文本内容匹配（不按编号）
   - 记录匹配到的段落位置
2. 执行替换：
   - 替换整个段落内容
   - 保留原段落的样式（字体、居中、缩进等）
   - 如果要保留部分原文，只替换指定文字
3. 替换后检查：
   - 段落数是否不变
   - 段落样式是否保持一致
   - 上下文逻辑是否通顺

**降AI替换规范**：
- 使用 `config/replacement_rules.json` 进行系统化替换
- 优先替换高风险词，再替换中风险词
- 每轮替换后用户测AI率，再决定是否继续

---

### format_modify 操作标准

**适用范围**：三线表、字体字号、行距、对齐等格式问题

**处理原则**：
1. 统一记录为"待格式处理"
2. 不在内容修改阶段处理
3. 在修改报告中注明：
   ```
   【格式类批注 - 待单独处理】
   - 位置：[段落]
   - 内容：[格式要求]
   - 建议：建议使用格式调整工具处理
   ```

**禁止行为**：
- 禁止在内容修改时调整格式
- 禁止修改字体、字号、行距
- 禁止修改表格结构

## 修改类型与执行流程

### 输入

读取阶段一输出的文件：
- 需求分析报告.md
- 资料核对清单.md
- 用户的文档文件

### 步骤1：提取修改要求

**支持三种输入，自动识别类型，统一输出JSON：**

| 输入类型 | 调用方式 | 说明 |
|---------|---------|------|
| Word批注 | `--input 论文.docx` | 文档自带导师批注 |
| 图片/截图 | `--input 修改要求截图.png` | 客户发的修改意见图片 |
| 纯文本 | `--text "1. xxx 2. xxx"` | 客户直接描述的修改要求 |

```bash
# 方式1：Word批注
python scripts/doc_analyzer.py --input 论文.docx --pretty

# 方式2：图片
python scripts/doc_analyzer.py --input 修改要求.png --pretty

# 方式3：纯文本
python scripts/doc_analyzer.py --text "1. 时间不对 2. 内容太少" --name 客户描述 --pretty
```

**统一输出格式：**
```json
{
  "source_type": "comments | image | text",
  "source_file": "文件名",
  "total": 12,
  "raw_text": "（图片OCR原始文本，供AI理解用）",
  "items": [
    {
      "id": "1",
      "type": "content_modify | content_add | content_delete | format_modify",
      "location": "批注所在段落 / 图片文件名 / 文本来源",
      "content": "修改要求原文",
      "author": "批注作者（仅comments有）"
    }
  ]
}
```

**type类型说明：**
- `content_modify`：内容修改（时间不对、太简单了）
- `content_add`：内容补充（需要添加xx）
- `content_delete`：内容删除（删除无意义代码）
- `format_modify`：格式调整（三线表、格式不对）

**第二步：逐条处理**

按 `items` 逐条处理，记录：
- 处理状态：已处理 / 待确认 / 无法处理
- 具体修改内容

**处理逻辑（核心）：**

| 修改类型 | 处理方式 |
|---------|---------|
| content_delete | 直接删除对应段落 |
| content_modify（轻微文字修改） | 直接替换文字 |
| content_modify（需要新内容） | 按反AI写作规范生成内容 → 替换 |
| content_add | 按反AI写作规范生成内容 → 插入到指定位置 |
| format_modify | 记录为"待格式处理" |

**生成反AI内容时必须遵守**：
1. 使用第一人称（"我"、"本次研究"、"在研究过程中"）
2. 加入研究过程中的思考、尝试、调整
3. 避免完美的线性逻辑，加入1-2个"合理的小波折"
4. 每个专业术语后紧跟具体细节
5. 长短句交替，不要统一句式
6. 禁止使用：深入研究、具有重要意义、由此可见、首先...其次...最后...

**第三步：输出修改报告**

```
【修改处理报告】
总数：12  已处理：X  待确认：X  无法处理：X

【1】位置：二〇二五年六月
类型：content_modify
内容：时间不对
处理：已将日期改为实际时间
状态：✅已处理
```

### 步骤4：内容调整

- 根据需求调整章节结构
- 补充缺失内容
- 修正逻辑问题
- **保持段落格式不变**

### 步骤5：验证与交接

1. 对比修改前后：段落数/节数/表格数是否一致
2. 如有差异 → 立即修正
3. 确认内容无误后：
   - 保存最终内容版文档
   - 通知用户：内容修改完成

---

## 迭代修改

如果客户对修改不满意：
1. 读取客户反馈
2. 定位不满意的具体段落
3. 重新修改该段落（不影响其他已确认的部分）
4. 再次验证格式一致性
5. 反复直到客户满意

---

## 输出成果

1. 修改后的文档（内容已确认版）
2. 批注处理报告（如有）
3. 修改记录（哪些段落被改动、原因是什么）
